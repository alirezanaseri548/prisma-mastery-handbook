۱. مقدمه‌ای بر Prisma

۱.۱. Prisma چیست؟

Prisma یک ORM مدرن برای Node.js و TypeScript است که کار با پایگاه‌داده‌های رابطه‌ای (مثل PostgreSQL، MySQL، SQL Server و SQLite) را ساده، امن و تایپ‌سیف می‌کند.

  • Prisma Client: لایبرری تایپ‌سیف برای کوئری‌زدن از داخل کد
  • Prisma Migrate: ابزار مدیریت مهاجرت (اسکیمای دیتابیس)
  • Prisma Studio: UI ساده برای مشاهده و مدیریت داده‌ها

۱.۲. چرا Prisma؟

  • تایپ‌سیف: خطاهای کوئری در زمان کامپایل مشخص می‌شوند.
  • Developer Experience عالی: تکمیل خودکار، اسکیما واضح، مستندات قوی.
  • پیاده‌سازی راحت روابط پیچیده: ۱-۱، ۱-n، n-n.
  • ادغام آسان با فریم‌ورک‌هایی مثل NestJS، Express، Next.js و …
نکته: Prisma جایگزین SQL نیست؛ در واقع یک لایه روی دیتابیس است که بسیاری از جزئیات کم‌اهمیت ولی تکراری را از دوش برنامه‌نویس برمی‌دارد.

۲. راه‌اندازی اولیه Prisma در پروژه

۲.۱. پیش‌نیازها

  • نصب Node.js (ترجیحاً نسخه LTS)
  • آشنایی اولیه با TypeScript (اختیاری ولی توصیه‌شده)
  • داشتن یک دیتابیس نمونه (مثلاً PostgreSQL یا SQLite)

۲.۲. ایجاد پروژه پایه

// پوشه پروژه
mkdir prisma-mastery
cd prisma-mastery

// مقداردهی اولیه Node.js
npm init -y

// نصب TypeScript و ابزارهای مرتبط (اختیاری ولی پیشنهاد‌شده)
npm install --save-dev typescript ts-node @types/node
npx tsc --init

۲.۳. نصب Prisma و Prisma Client

// نصب Prisma CLI و Prisma Client
npm install prisma --save-dev
npm install @prisma/client

// مقداردهی اولیه Prisma
npx prisma init

دستور npx prisma init یک پوشه prisma/ و فایل schema.prisma و همچنین یک فایل .env ایجاد می‌کند.

۲.۴. تنظیم اتصال به دیتابیس (DATABASE_URL)

در فایل .env آدرس اتصال به دیتابیس را تنظیم می‌کنیم:

DATABASE_URL="postgresql://user:password@localhost:5432/prisma_mastery?schema=public"

در فایل prisma/schema.prisma دیتابیس را مشخص می‌کنیم:

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

generator client {
  provider = "prisma-client-js"
}
هشدار: اطلاعات حساس (یوزرنیم، پسورد) را هرگز در گیت پابلیک قرار ندهید. فایل .env را در .gitignore قرار دهید.

۳. آشنایی با Prisma Schema

۳.۱. ساختار کلی schema.prisma

این فایل معمولا شامل سه بخش است:

  1. datasource: تنظیمات دیتابیس
  2. generator: خروجی‌های تولیدشده (مثلاً Prisma Client)
  3. model: تعریف مدل‌ها (که به جدول‌های دیتابیس نگاشت می‌شوند)

۳.۲. یک مدل ساده User

model User {
  id        Int      @id @default(autoincrement())
  email     String   @unique
  name      String?
  createdAt DateTime @default(now())
}

نکات:

  • @id فیلد را به عنوان کلید اصلی مشخص می‌کند.
  • @default(autoincrement()) مقدار خودکار را برای فیلدهای عددی ایجاد می‌کند.
  • @unique یعنی مقدار این فیلد در جدول یکتا است.
  • String? یعنی این فیلد Nullable (اختیاری) است.

۳.۳. انواع داده متداول در Prisma

  • Int ،BigInt
  • String
  • Boolean
  • DateTime
  • Float ،Decimal
  • Json
نکته: می‌توان برای هر دیتابیس، نوع‌های خاص خودش را هم استفاده کرد (مثل @db.VarChar(255)).

۴. مدل‌سازی رابطه ۱-چند (1-n)

۴.۱. سناریو: User و Post

یک کاربر (User) می‌تواند چند پست (Post) داشته باشد. یعنی رابطه‌ی ۱-چند بین User و Post.

۴.۲. تعریف مدل‌ها

model User {
  id    Int    @id @default(autoincrement())
  email String @unique
  name  String?

  posts Post[]  // 1-n: یک کاربر، چند پست
}

model Post {
  id        Int      @id @default(autoincrement())
  title     String
  content   String?
  published Boolean  @default(false)

  authorId  Int
  author    User     @relation(fields: [authorId], references: [id])
}

نکات مهم:

  • در مدل User یک آرایه Post[] داریم که سمت «۱» را نشان می‌دهد.
  • در مدل Post یک فیلد authorId به عنوان کلید خارجی داریم.
  • با @relation مشخص می‌کنیم که کدام فیلد کلید خارجی است و به کدام فیلد در مدل دیگر اشاره می‌کند.

۴.۳. خوانایی و نام‌گذاری روابط

در صورت داشتن چند رابطه بین دو مدل، باید نام رابطه را مشخص کنیم:

model User {
  id        Int     @id @default(autoincrement())
  email     String  @unique

  posts     Post[]  @relation("UserPosts")
  comments  Comment[] @relation("UserComments")
}

model Post {
  id        Int     @id @default(autoincrement())
  title     String

  authorId  Int
  author    User    @relation("UserPosts", fields: [authorId], references: [id])

  comments  Comment[]
}

model Comment {
  id        Int     @id @default(autoincrement())
  text      String

  authorId  Int
  author    User    @relation("UserComments", fields: [authorId], references: [id])

  postId    Int
  post      Post    @relation(fields: [postId], references: [id])
}
نکته: نام‌گذاری روابط کمک می‌کند از خطاهای «ambiguous relation» در زمان prisma generate یا prisma migrate جلوگیری شود.

۵. مدل‌سازی رابطه چند-چند (n-n)

۵.۱. سناریو: Post و Tag

هر پست می‌تواند چند تگ داشته باشد، و هر تگ می‌تواند روی چند پست اعمال شود: این یک رابطه‌ی n-n است.

۵.۲. رابطه چند-چند ضمنی (Implicit Many-to-Many)

model Post {
  id        Int     @id @default(autoincrement())
  title     String

  tags      Tag[]   // n-n
}

model Tag {
  id     Int     @id @default(autoincrement())
  name   String  @unique

  posts  Post[]
}

Prisma به‌صورت خودکار یک جدول واسط (join table) ایجاد می‌کند (مثلاً _PostToTag) که دو کلید خارجی دارد.

۵.۳. رابطه چند-چند صریح (Explicit Many-to-Many)

وقتی بخواهیم روی رابطه، فیلد اضافه داشته باشیم (مثلاً تاریخ ایجاد، یا وزن/اولویت)، باید جدول واسط را خودمان تعریف کنیم: 

model Post {
  id         Int           @id @default(autoincrement())
  title      String

  postTags   PostTag[]
}

model Tag {
  id         Int           @id @default(autoincrement())
  name       String        @unique

  postTags   PostTag[]
}

model PostTag {
  id        Int     @id @default(autoincrement())
  postId    Int
  tagId     Int

  // فیلدهای اضافه روی رابطه
  createdAt DateTime @default(now())
  weight    Int      @default(1)

  post      Post   @relation(fields: [postId], references: [id])
  tag       Tag    @relation(fields: [tagId], references: [id])

  @@unique([postId, tagId])
}
نکته: در پروژه‌های واقعی، رابطه n-n صریح بسیار رایج است؛ چون معمولاً روی ارتباط هم متادیتا (مثل زمان برچسب‌گذاری) نیاز داریم.

۶. مهاجرت‌ها (Migrations) در Prisma

۶.۱. ایجاد اولین مایگریشن

پس از تعریف مدل‌ها در schema.prisma، با دستور زیر اولین مایگریشن را ایجاد می‌کنیم:

npx prisma migrate dev --name init

این دستور:

  • یک پوشه prisma/migrations می‌سازد.
  • SQL متناظر با مدل‌ها را تولید و روی دیتابیس اعمال می‌کند.
  • Prisma Client را دوباره تولید می‌کند.

۶.۲. به‌روزرسانی اسکیمای دیتابیس

فرض کنید فیلد جدیدی به مدل Post اضافه می‌کنیم:

model Post {
  id        Int      @id @default(autoincrement())
  title     String
  content   String?
  published Boolean  @default(false)

  // فیلد جدید:
  slug      String   @unique
}

سپس مایگریشن جدید اجرا می‌کنیم:

npx prisma migrate dev --name add-slug-to-post

۶.۳. Prisma Studio برای بررسی داده‌ها

برای بررسی سریع داده‌ها و تست CRUD، از Prisma Studio استفاده می‌کنیم: 

npx prisma studio

یک UI تحت وب باز می‌شود که در آن می‌توان رکوردها را مشاهده/ایجاد/ویرایش کرد.

۷. پیاده‌سازی CRUD با Prisma Client

۷.۱. تولید Prisma Client

هر بار که npx prisma migrate dev را اجرا می‌کنیم، به‌طور خودکار Prisma Client هم regenerate می‌شود. برای تولید دستی: 

npx prisma generate

۷.۲. ایجاد یک فایل entry point

برای تست ساده CRUD، یک فایل src/index.ts (یا index.js) می‌سازیم:

// src/index.ts
import { PrismaClient } from '@prisma/client'

const prisma = new PrismaClient()

async function main() {
  // TODO: CRUD operations
}

main()
  .catch(e => {
    console.error(e)
    process.exit(1)
  })
  .finally(async () => {
    await prisma.$disconnect()
  })

۷.۳. Create (ایجاد)

// ایجاد یک کاربر جدید
const user = await prisma.user.create({
  data: {
    email: 'user1@example.com',
    name: 'User One',
  },
})

// ایجاد یک پست مرتبط با این کاربر (۱-n)
const post = await prisma.post.create({
  data: {
    title: 'اولین پست با Prisma',
    content: 'متن نمونه...',
    author: {
      connect: { id: user.id },
    },
  },
})

۷.۴. Read (خواندن)

// یافتن یک کاربر با ایمیل خاص
const user = await prisma.user.findUnique({
  where: { email: 'user1@example.com' },
})

// یافتن یک کاربر همراه با پست‌ها
const userWithPosts = await prisma.user.findUnique({
  where: { id: 1 },
  include: {
    posts: true,
  },
})

// دریافت لیست پست‌ها با فیلتر و مرتب‌سازی
const posts = await prisma.post.findMany({
  where: {
    published: true,
    title: { contains: 'Prisma' },
  },
  orderBy: {
    createdAt: 'desc',
  },
  take: 10,
})

۷.۵. Update (به‌روزرسانی)

// به‌روزرسانی یک پست
const updatedPost = await prisma.post.update({
  where: { id: 1 },
  data: {
    title: 'عنوان جدید برای پست',
    published: true,
  },
})

۷.۶. Delete (حذف)

// حذف یک پست
const deletedPost = await prisma.post.delete({
  where: { id: 1 },
})
نکته: اگر روابط و محدودیت‌های foreign key به‌درستی تنظیم شده باشند، هنگام حذف رکورد، ممکن است خطا دریافت کنید (مثلاً اگر رکورد فرزند هنوز موجود باشد). برای این موارد باید استراتژی حذف (cascade / restrict) را در دیتابیس و مدل‌ها مشخص کنید.

۸. کوئری‌های پیشرفته و چند مدل همزمان

۸.۱. nested write (نوشتن تو در تو)

در Prisma می‌توان در یک عملیات، هم‌زمان چند مدل مرتبط را ذخیره کرد:

// ایجاد کاربر به همراه پست و تگ‌های آن (رابطه n-n)
const createdUser = await prisma.user.create({
  data: {
    email: 'nested@example.com',
    name: 'Nested Example',
    posts: {
      create: [
        {
          title: 'پست همراه با تگ',
          content: '...',
          tags: {
            create: [
              { name: 'prisma' },
              { name: 'backend' },
            ],
          },
        },
      ],
    },
  },
  include: {
    posts: {
      include: { tags: true },
    },
  },
})

۸.۲. Selection و Projection

برای کاهش حجم داده، می‌توانیم فقط فیلدهای مورد نیاز را انتخاب کنیم:

const posts = await prisma.post.findMany({
  select: {
    id: true,
    title: true,
    author: {
      select: {
        id: true,
        email: true,
      },
    },
  },
})

۸.۳. Pagination ساده

const page = 1
const pageSize = 10

const posts = await prisma.post.findMany({
  skip: (page - 1) * pageSize,
  take: pageSize,
  orderBy: { createdAt: 'desc' },
})

۹. الگوهای متداول در Backend با Prisma

۹.۱. ساختار لایه‌ای (Layered Architecture)

  • Layer Controller/Route: دریافت درخواست HTTP، اعتبارسنجی اولیه
  • Service Layer: منطق کسب‌وکار، استفاده از Prisma Client
  • Data Access Layer (اختیاری): wrapperهای مشخص برای عملیات CRUD

۹.۲. مدیریت lifecycle PrismaClient

  • در اکثر پروژه‌ها یک PrismaClient singleton ساخته و در کل برنامه استفاده می‌شود.
  • در فریم‌ورک‌هایی مانند NestJS، PrismaClient به‌صورت یک Provider ثبت می‌شود.

۹.۳. نکات عملکردی (Performance)

  • استفاده از select و include به‌صورت مینیمال
  • پرهیز از n+1 query با استفاده از include مناسب
  • استفاده از indexها در دیتابیس برای فیلترهای پرتکرار
نکته: در پروژه‌های بزرگ، طرح‌ریزی دقیق روابط (به‌خصوص ۱-n و n-n) و طراحی اسکیمای درست، تأثیر مستقیم روی عملکرد و سادگی توسعه دارد.

۱۰. چک‌لیست «Prisma Ready» برای پروژه

برای اطمینان از اینکه پروژه‌ی شما از نظر Prisma آماده است، آیتم‌های زیر را بررسی کنید:

  • ☑ فایل .env تنظیم و DATABASE_URL صحیح است.
  • ☑ فایل schema.prisma حداقل شامل یک مدل پایه (مثلاً User) است.
  • ☑ روابط ۱-n بین مدل‌های اصلی (مثل User-Post) به‌درستی تعریف شده‌اند.
  • ☑ روابط n-n مورد نیاز، با توجه به نیاز پروژه (implicit یا explicit) طراحی شده‌اند.
  • ☑ حداقل یک مایگریشن (prisma migrate dev) با موفقیت اجرا شده است.
  • ☑ Prisma Client بدون خطا تولید می‌شود (prisma generate).
  • ☑ تست CRUD برای مهم‌ترین مدل‌ها انجام شده است.
  • ☑ از Prisma Studio برای بررسی داده‌ها استفاده و ساختار روابط را تأیید کرده‌اید.
  • ☑ الگوی مناسب برای استفاده از PrismaClient در کل پروژه مشخص شده است.

با انجام این چک‌لیست، می‌توانید مطمئن باشید که پروژه‌ی Backend شما از نظر کار با Prisma، برای توسعه‌ی سریع و افزودن فیچرهای جدید آماده است.
